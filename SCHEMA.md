## Entity-Relationship Overview

```
candidates
    │
    │  (one candidate may receive many offers)
    │
    ▼
offers ──────────────────────────────────────────┐
    │                                             │
    │  (one offer → many events)                 │
    ▼                                             │
offer_events                                      │
    [offer_generated]                             │
    [offer_sent]                                  │
    [offer_opened]                                │
    [offer_signed]  ──────────────────────────────┤
    [offer_rejected]                              │
                                                  │
                              signatures ◀────────┤
                                  │               │
                                  ▼               │
                         verification_logs ◀──────┘
```

**Cardinalities:**

| Relationship | Type |
|---|---|
| candidate → offers | 1 : many |
| offer → offer_events | 1 : many |
| offer → signatures | 1 : 0..1 (at most one signature per offer) |
| offer → verification_logs | 1 : many (re-checks allowed) |
| signatures → verification_logs | 1 : many |

## Tables

### `candidates`

Stores one record per job candidate who may receive an offer.
Created upstream in the hiring pipeline; referenced by `offers`.

| Column | Type | Nullable | Constraint | Description |
|--------|------|----------|------------|-------------|
| `candidate_id` | TEXT | NO | PRIMARY KEY | UUID v4 — unique candidate identifier |
| `full_name` | TEXT | NO | NOT NULL | Candidate's full legal name |
| `email` | TEXT | NO | NOT NULL, UNIQUE | Contact email — must be unique across all candidates |
| `phone` | TEXT | YES | — | Contact phone number (max 15 chars) |
| `role_applied` | TEXT | NO | NOT NULL | Job title the candidate applied for |
| `department` | TEXT | NO | NOT NULL | Hiring department (Engineering, Product, etc.) |
| `created_at` | TEXT | NO | NOT NULL | ISO-8601 UTC timestamp of candidate record creation |
| `source_channel` | TEXT | NO | NOT NULL | Acquisition channel: `linkedin` \| `referral` \| `company_portal` \| `naukri` \| `internshala` |

**Indexes:**
- `PRIMARY KEY (candidate_id)` — implicit unique index
- `UNIQUE (email)` — enforced at DB level

**Sample row:**
```
candidate_id  : "a1b2c3d4-..."
full_name     : "Priya Sharma"
email         : "priya.sharma@example.com"
phone         : "+91-9876543210"
role_applied  : "Data Analyst"
department    : "Data"
created_at    : "2026-04-01T09:30:00Z"
source_channel: "linkedin"
```

### `offers`

One row per offer document generated by HR-ops.
The `status` column reflects the **latest known state** of the offer;
the canonical history lives in `offer_events`.

| Column | Type | Nullable | Constraint | Description |
|--------|------|----------|------------|-------------|
| `offer_id` | TEXT | NO | PRIMARY KEY | UUID v4 — unique offer identifier |
| `candidate_id` | TEXT | NO | NOT NULL, FK → candidates | Links offer to candidate |
| `generated_at` | TEXT | NO | NOT NULL | ISO-8601 UTC — when offer document was created |
| `sent_at` | TEXT | YES | — | ISO-8601 UTC — when offer was dispatched; NULL until sent |
| `status` | TEXT | NO | NOT NULL, CHECK | Current offer status (see allowed values below) |
| `salary_inr` | INTEGER | NO | NOT NULL | Offered annual salary in Indian Rupees |
| `role_title` | TEXT | NO | NOT NULL | Specific role title on the offer letter |
| `expiry_at` | TEXT | NO | NOT NULL | ISO-8601 UTC — offer validity deadline |

**Status CHECK constraint — allowed values:**

| Status | Meaning |
|--------|---------|
| `generated` | Offer created; not yet sent |
| `sent` | Dispatched to candidate |
| `viewed` | Candidate opened the offer |
| `signed` | Candidate completed e-sign |
| `rejected` | Candidate explicitly declined |

**Indexes:**
```sql
idx_offers_candidate  ON offers (candidate_id)
idx_offers_status     ON offers (status)
idx_offers_generated  ON offers (generated_at)
```

**Foreign keys:**
```sql
FOREIGN KEY (candidate_id) REFERENCES candidates (candidate_id)
```

### `offer_events`

**Immutable event log** — the single source of truth for the offer
lifecycle. Every state change appends a new row; no rows are updated
or deleted. All funnel metrics are computed from this table.

| Column | Type | Nullable | Constraint | Description |
|--------|------|----------|------------|-------------|
| `event_id` | TEXT | NO | PRIMARY KEY | UUID v4 — unique event identifier |
| `offer_id` | TEXT | NO | NOT NULL, FK → offers | Links event to its offer |
| `event_name` | TEXT | NO | NOT NULL, CHECK | Event type (see allowed values below) |
| `timestamp` | TEXT | NO | NOT NULL | ISO-8601 UTC — when event occurred |
| `actor` | TEXT | YES | — | Who triggered the event: `system` \| `candidate` \| `hr_ops` |
| `metadata` | TEXT | YES | — | JSON blob with event-specific context (device, channel, reason) |

**Event CHECK constraint — allowed values:**

| event_name | Triggered by | Meaning |
|------------|-------------|---------|
| `offer_generated` | system | Offer document created after HR-ops approval |
| `offer_sent` | system | Offer dispatched via email/SMS |
| `offer_opened` | candidate | Candidate opened the offer link |
| `offer_signed` | candidate | Candidate completed the e-sign flow |
| `offer_rejected` | candidate | Candidate explicitly declined the offer |

**Metadata examples:**
```json
// offer_generated
{"trigger": "hr_ops_approval"}

// offer_sent
{"channel": "email", "provider": "sendgrid"}

// offer_opened
{"device": "mobile", "ip_hash": "a3f9b2c1..."}

// offer_signed
{"method": "aadhaar_otp"}

// offer_rejected
{"reason": "compensation_low"}
```

**Indexes:**
```sql
idx_events_offer  ON offer_events (offer_id)
idx_events_name   ON offer_events (event_name)
idx_events_ts     ON offer_events (timestamp)
```

**Design note:** This table is append-only. Never UPDATE or DELETE rows.
Metrics that depend on the "latest" event use window functions or
correlated subqueries rather than relying on `offers.status`.

### `signatures`

One row per completed e-sign transaction.
A signature row is created only after `offer_signed` fires.
Enforces `UNIQUE(offer_id)` — one legal signature per offer.

| Column | Type | Nullable | Constraint | Description |
|--------|------|----------|------------|-------------|
| `sign_id` | TEXT | NO | PRIMARY KEY | UUID v4 — unique signature identifier |
| `offer_id` | TEXT | NO | NOT NULL, UNIQUE, FK → offers | Links to signed offer; UNIQUE enforces one-sig-per-offer |
| `provider` | TEXT | NO | NOT NULL | E-sign provider: `leegality` \| `digio` \| `docusign` |
| `signed_at` | TEXT | NO | NOT NULL | ISO-8601 UTC — timestamp of signature completion |
| `verification_status` | TEXT | NO | NOT NULL, CHECK | Tamper-check outcome (see below) |
| `signer_ip_hash` | TEXT | YES | — | SHA-256 hash of signer IP (never store raw IP) |
| `document_hash` | TEXT | YES | — | SHA-256 hash of signed PDF — tamper detection |

**Verification status CHECK constraint:**

| Value | Meaning |
|-------|---------|
| `pending` | Verification not yet attempted |
| `verified` | Provider confirmed signature is authentic |
| `failed` | Verification check failed (see verification_logs for detail) |

**Indexes:**
```sql
idx_signatures_offer  ON signatures (offer_id)
```

**Security note:** `document_hash` enables tamper-evidence. If the stored
hash does not match a re-hash of the PDF, the document has been altered.
`signer_ip_hash` preserves audit capability without storing PII.

### `verification_logs`

Audit trail of every verification attempt against a signature.
Multiple rows per offer are allowed (re-checks after failures).

| Column | Type | Nullable | Constraint | Description |
|--------|------|----------|------------|-------------|
| `verification_id` | TEXT | NO | PRIMARY KEY | UUID v4 — unique log entry identifier |
| `offer_id` | TEXT | NO | NOT NULL, FK → offers | Links verification attempt to offer |
| `check_time` | TEXT | NO | NOT NULL | ISO-8601 UTC — when verification was attempted |
| `result` | TEXT | NO | NOT NULL, CHECK | Outcome: `pass` \| `fail` \| `pending` |
| `check_type` | TEXT | NO | NOT NULL | Method: `hash_check` \| `provider_webhook` \| `manual` |
| `error_message` | TEXT | YES | — | Human-readable failure reason; NULL on pass |

**Result CHECK constraint:**

| Value | Meaning |
|-------|---------|
| `pass` | Verification succeeded — signature is authentic |
| `fail` | Verification failed — see `error_message` |
| `pending` | Verification queued but not yet complete |

**Check type values:**

| check_type | Description |
|------------|-------------|
| `hash_check` | Local SHA-256 comparison of document hash |
| `provider_webhook` | Real-time callback from e-sign provider |
| `manual` | HR-ops manual review after automated failure |

**Indexes:**
```sql
idx_verif_offer  ON verification_logs (offer_id)
idx_verif_time   ON verification_logs (check_time)
```

## Timestamp Convention

All timestamps are stored as **ISO-8601 UTC strings**:
```
2026-06-01T14:30:00Z
```

SQLite functions used in queries:
```sql
JULIANDAY(timestamp)           -- for arithmetic (time differences)
DATE(timestamp)                -- for daily grouping
STRFTIME('%Y-W%W', timestamp)  -- for weekly grouping
datetime('now')                -- current UTC time
datetime('now', '-30 days')    -- 30 days ago
```
